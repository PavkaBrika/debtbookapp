## Design Context

### Users
- Primary audience: a single person using the app for personal tracking.
- Context: quick, frequent check-ins on a phone; users want clarity without friction or anxiety.
- Core jobs-to-be-done across the app:
  - Track **debts** per person (add/edit, review history, share/export when needed).
  - Track **finances** (income/expenses) by category and time intervals; review history fast (scan → understand → optionally act).
  - Track **savings goals** with deposits; see progress and remaining amount/time.
  - Keep data safe: optional **PIN/biometrics**, optional **Google Drive sync**.

### Brand Personality
- Tone: calm, trustworthy, low-anxiety.
- The interface should feel steady and reassuring, not “aggressive” or gamified.

### Aesthetic Direction
- Direction: clean, functional UI with a strong “system” feel.
- UI stack reality: the project is in an **XML → Compose migration** phase:
  - Legacy screens: Views + `Theme.MaterialComponents.DayNight` (Material Components).
  - New screens/components: Compose + Material3.
  - Goal: users should perceive it as one product (shared tokens, patterns, copy tone).
- Theme support: light and dark are equally important.
- Palette intent: neutral base with a restrained mint/teal accent used purposefully, not decoratively.

### Design Principles
1. Optimize for scanability: clear hierarchy, predictable structure, fast parsing of numbers and dates.
2. Reduce cognitive load: show the minimum by default; reveal secondary actions progressively.
3. Calm confidence: avoid visual noise and “alarm” colors unless communicating risk/errors.
4. Consistency over novelty: Material3 patterns, consistent spacing, typography, and component behavior across features.
5. Safe actions: destructive operations must be clearly separated, reversible where possible (undo), and never accidental.
6. State clarity everywhere: loading/empty/error/success must be explicit and non-blaming.
7. Migration discipline: when moving a screen to Compose, preserve the user’s mental model (information order, naming, primary actions) even if components change.
